Spring自调用事务失效 ​

Methods with Spring proxy should not be called via "this"

最近修复 sonar 代码检查的时候发现一个事务的报错，大意就是说不要直接调用当前实例的方法，而是要通过间接调用，否则事务可能会失效，原代码如下：

java
public class TestImpl extends Test {
    
    @Transactional(rollbackFor = {Exception.class})
    @Override
    public void a() {
        this.b();
    }
    
    @Transactional(rollbackFor = {Exception.class})
    @Override
    public void b(){
        // do something
    }
    
}

sonar 建议我们改成以下方式：

java
public class TestImpl extends Test {
    @Resource
    private Test test;

    @Transactional(rollbackFor = {Exception.class})
    @Override
    public void a(){
        test.b();
    }

    @Transactional(rollbackFor = {Exception.class})
    @Override
    public void b(){
        // do something
    }

}

为什么要改成第三方调用呢？这不是属于脱裤子放屁么？然后我去查了一下原因，发现这与 @Transactional 注解生效的时机有关系，我们知道当注解定义后，需要有一个地方进行处理才能有作用，在 Spring 中则是把每一个由它管理的类都生成了一个 Proxy 类，然后在 Proxy 类中去判断和处理注解，最终再调用对应的方法，用伪代码解释就是：

java
public class TestProxy {
    
    private Test test;
    
    public void a(){
        // 判断是否有 @Transactional 注解，如果有则开启事务
        // 开始事务
        test.a();
        // 结束事务
    }
    
    public void b(){
        // 判断是否有 @Transactional 注解，如果有则开启事务
        // 开始事务
        test.b();
        // 结束事务
    }
    
}

然后再把我们写的代码代入进去，当调用 TestImpl.A() 时，本质是调用 TestProxy.A()，然后 TestProxy.A() 中发现我们有 @Transactional 注解，于是开启事务，接着再调用真实的对象 test.a() , 接着在 test.a() 中调用了 this.b() ，注意此时 this.b() 是直接调用的 Test 类，而不是 TestProxy 类，所以并不会走对应处理注解的流程，此时定义的注解就不会生效，因此 sonar 建议我们间接调用，本意就是想拿到 TestProxy 再去调用 b 方法，换成其伪代码如下：

java
public class TestImpl extends Test {
    @Resource
    private TestProxy testProxy;

    @Transactional(rollbackFor = {Exception.class})
    @Override
    public void a(){
        testProxy.b();
    }

    @Transactional(rollbackFor = {Exception.class})
    @Override
    public void b(){
        // do something
    }

}

最后再思考一下，为什么当 @Transactional 注解遇到 private 修饰符会失效，原因也很简单，Spring AOP 动态代理生成的代理默认使用的是 CGLIB，而 CGLIB 因为采用的是继承的方式代理对象，所以根据继承的约束，无法代理 final 和 private 修饰的属性或方法。

AspectJ 是一个 AOP 框架，用于实现 AOP，相同地位的则是 Spring AOP，但他们两个不同点在于：

Spring AOP 属于动态代理的方式来实现 AOP，在运行时生成 Proxy 对象，依靠 CGLIB 和 JDK 动态代理的方式来实现(JDK 依赖接口，CGLIB不依赖接口)。

AspectJ 属于静态代理的方式来实现 AOP，在编译时生成 Proxy 对象，所以比 Spring AOP 更强大。

所以再回到上面的问题，如果我们将 Spring AOP 的底层编译器和织入器改成 Aspectj，这样就可以代理 final 和 private 了。注：Spring AOP 只是采用了 Aspectj 包提供的注解而已。
