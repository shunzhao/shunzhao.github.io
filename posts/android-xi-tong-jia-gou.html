Android系统架构 ​

Android 的系统架构大致可以分为四层架构：Linux内核层、系统运行库层（虚拟机层）、应用框架层(api层)和应用层。

Linux内核层 ​

提供了硬件层面的驱动

系统运行库层（虚拟机层） ​

一些C/C++库为Android系统提供了主要的特性支持，还包括运行库 Android运行时库中还包含了Dalvik虚拟机（5.0系统之后改为ART运行环境），它使得每一个Android应用都能运行在独立的进程当中，并且拥有一个自己的Dalvik虚拟机实例

应用框架层(api层) ​

这一层主要提供了构建应用程序时可能用到的各种API

四大组件 ​

Android系统四大组件分别是活动（Activity）、服务（Service）、广播接收器（Broadcast Receiver）和内容提供器（Content Provider）。 Activity 是所有Android应用程序的门面 Service 后台默默地运行 BroadcastReceiver 允许你的应用接收来自各处的广播消息 ContentProvider则为应用程序之间共享数据

Activity 是Android系统提供的一个 Activity 基类，我们项目中所有的 Activity 都必须继承它或者它的子类才能拥有 Activity 的特性（AppCompatActivity是Activity的子类

应用程序模块和库模块的最大区别在于，一个是可以直接运行的，一个只能作为代码库依附于别的应用程序模块来运行。

active 之间的跳转 ​

Intent是Android程序中各组件之间进行交互的一种重要方式，它不仅可以指明当前组件想要执行的动作，还可以在不同组件之间传递数据。Intent一般可被用于启动活动、启动服务以及发送广播等场景

显示跳转： final Intent intent = new Intent(FirstActivity.this, SecondActivity.class); startActivity(intent);

隐式跳转： 隐式并不明确指出我们想要启动哪一个活动，而是指定了一系列更为抽象的action和category等信息，通过匹配action和category的方式，让系统决定启动哪个 activity ， 操作步骤：在 AndroidManifest.xml 文件中对应的 activity 标签下，添加 <intent-filter> 子标签，然后再改子标签下添加 <actiion/> 或者 <category/>，其中 category标签有一个默认值:android.intent.category.DEFAULT

显示和隐式的区别在哪里？ 答：使用隐式Intent，我们不仅可以启动自己程序内的活动，还可以启动其他程序的活动，这使得Android多个应用程序之间的功能共享成为了可能，说的直白点就是手机中的打开方式，比如点击了一个 text 系统会让你选择用哪个程序打卡，这个功能就是这样通过隐式跳转实现 具体的实现步骤，必须通过 <data/> 标签只有<data>标签中指定的内容和Intent中携带的Data完全一致时，当前活动才能够响应该Intent,

总结：<intent-filter />总用有三个元素分别是：<action/> <category/> <data/>

活动的生命周期: 什么是返回栈？ 返回栈是指每当 start 一个 activity 时，就会把该 activity 放入栈中，当一个 activity 被销毁时(back键或者finish)，就会弹出栈，显示下一个 activity，因此用户看到的永远时栈顶的 activity Android是使用任务（Task）来管理活动的一个任务就是一组存放在栈里的活动的集合，这个栈也被称作返回栈（Back Stack）

活动状态： 返回栈的栈顶时，即用户当前见到的 activity ，系统不会回收 暂停状态： 不处于栈顶，但仍然可见的状态，比如弹窗后，背景 activity， 还是可见，系统非必要不回收 停止状态： 不处于栈顶，并不可见状态，系统可能会回收 销毁状态： 不在返回栈中，系统一定会回收 以上类似于 java 中的引用，活动状态相当于强引用、暂停状态相当于软引用，停止状态相当于弱引用、销毁状态相当于虚引用

活动的生存期 ​

onCreate activity 被创建时执行 onStart() activity 可见或跳转时执行 onResume() activity 可交互时执行 onPause() activity 启动或恢复另一个 activity onStop() activity 变不可见时执行 onDestroy() activity 销毁时执行

3种生存期 完整生存期 一般在会在onCreate()方法中完成各种初始化操作，而在onDestroy()方法中完成释放内存的操作 可见生存期 onStart()方法中对资源进行加载，而在onStop()方法中对资源进行释放，从而保证处于停止状态的活动不会占用过多内存 前台生存期

当 activity 被回收时，再次 back 时，会执行 oncreate方法，此时为了用户体验，可以重写 onSaveInstanceState() 方法来保存数据

activity 启动模式 ​

启动模式一共有4种，分别是standard、singleTop、singleTask和singleInstance standard 原型模式 singleTop 栈顶模式 singleTask 单例模式，在整个返回栈中只有一个 activity singleInstance 返回一个新的返回栈以便整个系统内只有一个实例

假设有 oneActivity、twoActivity、threeActivity ，其中 twoactivity 启动模式为 singleInstance ，oneActivity 中的按钮启动 twoActivity，然后 twoActivity 中的按钮启动 threeActivity，其 back 时的顺序是什么？ 答 threeActivity -> oneActivity -> twoActivity 因为，threeActivity 和 oneActivity 的返回栈是同一个

最佳实践： 通过实现一个 baseactivity 的方式实现layout获取对应的 activity 通过编写一个 startActivity 的方法来实现必填参数的传递 通过再 baseActivity 的 onCreate 方法和 onDestory 方法中添加一个保存 activityList 集合实现一次关闭所有的 activity

本地广播和全局广播的区别 ​

service ​

Service 是 Android 中实现程序后台运行的解决方案，它非常适合去执行那些不需要和用户交互而且还要求长期运行的任务。服务的运行不依赖于任何用户界面，即使程序被切换到后台，或者用户打开了另外一个应用程序，服务仍然能够保持正常运行,不过需要注意的是，服务并不是运行在一个独立的进程当中的，而是依赖于创建服务时所在的应用程序进程。当某个应用程序进程被杀掉时，所有依赖于该进程的服务也会停止运行

在子线程中操作 UI ​

Android是不允许在子线程中进行UI操作的，实现这一功能，可以使用一下两种方案：

解析异步消息处理机制 ​

通过在主线程中创建一个消息中心，然后在子线程中发送一个消息，这样主线程在接受到后就会处理对应逻辑，这样就实现了子线程操作 UI

安卓异步消息模型 ​

Message ​

Message是在线程之间传递的消息，它可以在内部携带少量的信息，用于在不同线程之间交换数据

Handler（已废除） ​

Handler顾名思义也就是处理者的意思，它主要是用于发送和处理消息的

MessageQueue ​

MessageQueue是消息队列的意思，它主要用于存放所有通过Handler发送的消息。这部分消息会一直存在于消息队列中，等待被处理。每个线程中只会有一个MessageQueue对象

Looper ​

Looper是每个线程中的MessageQueue的管家，调用Looper的loop()方法后，就会进入到一个无限循环当中，然后每当发现MessageQueue中存在一条消息，就会将它取出，并传递到Handler的handleMessage()方法中 每个线程中也只会有一个Looper对象

这种写法问题很大，首先需要拥有一个成员 handler、

停止服务 ​

我们是完全有可能对一个Service既调用了startService()方法，又 调用了bindService()方法的，在这种情况下该如何让Service销毁呢？根据Android系统的 机制，一个Service只要被启动或者被绑定了之后，就会处于运行状态，必须要让以上两种条件 同时不满足，Service才能被销毁。所以，这种情况下要同时调用stopService()和 unbindService()方法，onDestroy()方法才会执行

从Android 8.0系统开始，应用的后台功能被大 幅削减。现在只有当应用保持在前台可见状态的情况下，Service才能保证稳定运行，一旦应用 进入后台之后，Service随时都有可能被系统回收

前台Service ​

前台服务，减少 app 被杀死的可能性，创建一个前台 service 的方法也很简单，本质上其实就是一个常驻的通知，所以创建方法也和通知是一样的， 只是在最后使用 startForeground() 方法启动，参数一样。 需要注意的是在 9.0 之后需要在 manifest 文件中申明权限

自动停止服务 ​

为了可以简单地创建一个异步的、会自动停止的服务，Android专门提供了一个IntentService类， 使用方法： 继承 IntentService 类，实现 现onHandleIntent 方法，这个方法默认在 子线程 中执行，启动方式和普通的 service 没有区别
